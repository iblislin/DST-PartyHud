# DST dependency surface ‚Äî PartyHud 2026

This catalogues every Don't Starve Together engine API, asset, build, event, netvar, TUNING value and
behavioural assumption the mod relies on, cross-checked against the decompiled game source in
`/home/iblis/code/dst/dst-scripts/scripts`. It is the diff target for the `dst-game-update-compat` SOP:
when DST (the game) is patched, each row's **Verify** check is re-run against the updated `dst-scripts`
to catch a game-side change BEFORE it silently breaks the mod ‚Äî the way the original PartyHUD broke when
the `health` build was renamed to `status_health`.

Generated from the mod at master HEAD via a fan-out map of mod-code ‚Üî dst-scripts (175 raw touchpoints ‚Üí
137 deduped rows). MAINTAIN IT: add a row whenever a feature introduces a new engine dependency (e.g.
v2026.13 adds `temperature.overheattemp` / `.rate` / `GetCurrent`, moisture, and `TUNING.OVERHEAT_TEMP`).

> **Note on character-specific thresholds:** as of this map, freeze is hardcoded `temperature.current < 0`
> and overheat is `current > overheattemp` (default `TUNING.OVERHEAT_TEMP = 70`), universal across
> characters ‚Äî only insulation (rate) and hurt-rate differ per character/item. If a future DST update or
> mod makes the freeze/overheat THRESHOLD character-specific, the temperature-danger logic must switch
> from fixed constants to each player's broadcast `temperature.overheattemp`.

## How to use on a DST update
1. Refresh the local `dst-scripts` checkout to the new game build.
2. For each row below, run its **Verify after a DST update** check against the refreshed `dst-scripts`.
3. Flag any mismatch (renamed symbol, changed signature/field, changed TUNING value, new character-specific
   threshold, removed asset). A mismatch = a likely break; read its **Break risk & symptom**.
4. After the static diff: run busted (`luajit spec/run_local.lua`) + luacheck, then an in-engine
   load-smoke on the modtest server WITH a player connected (a no-player smoke misses tick-path breaks).
